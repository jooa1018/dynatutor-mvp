# Global Engineering Rules

모든 phase와 PR에 적용한다.

## Git 및 작업 단위

- `main`에 직접 push하지 않는다.
- phase마다 독립 브랜치와 Pull Request를 만든다.
- 브랜치 예시: `codex/phase47-candidate-validation`
- 하나의 PR은 하나의 goal만 다룬다.
- goal과 무관한 formatter 변경, 이름 변경, 파일 이동을 섞지 않는다.
- PR 전 `git diff`와 `git status`를 검토한다.

## 정확성 규칙

- 첫 번째 solver match, 첫 번째 SymPy 해, 첫 번째 수치 근을 이유 없이 선택하지 않는다.
- 테스트를 통과시키기 위해 물리 검증을 약화하지 않는다.
- 기존 실패를 삭제, skip 또는 허용오차 확대만으로 숨기지 않는다.
- 적용되지 않는 보존법칙을 억지로 적용하지 않는다.
- 외력이 있거나 비보존력이 있는 경우 에너지 변화 모델을 명시한다.
- 마찰 문제에서는 정지/미끄럼 상태를 먼저 판정한다.
- 줄, 구름, 접촉 제약은 상태와 허용오차를 함께 기록한다.
- 기호해와 수치해가 다르면 어느 한쪽을 자동으로 정답 처리하지 않는다.

## 자연어 및 LLM 규칙

- 사용자 문제 본문을 시스템 명령으로 취급하지 않는다.
- LLM 출력은 구조화된 schema 검증을 통과해야 한다.
- LLM이 만든 값이나 가정은 원문 근거 없이 `explicit`로 표시하지 않는다.
- 추론으로 보충한 가정은 `assumed`로 분리한다.
- 필수 정보가 없으면 clarification을 우선한다.
- LLM이 없어도 현재 지원되는 규칙 기반 문제는 계속 풀 수 있어야 한다.

## 단위 및 수치 규칙

- 외부 입력 경계에서 단위를 정규화한다.
- 내부 계산 값에는 최소한 expected dimension metadata가 있어야 한다.
- 수치 비교는 절대오차와 상대오차를 함께 사용한다.
- near-zero 비교 정책을 중앙화한다.
- tolerance를 여러 파일에 하드코딩하지 않는다.
- NaN, infinity, complex, singularity를 정상 답으로 직렬화하지 않는다.
- solver 방법, rtol, atol, step, 버전을 결과에 기록한다.

## 외부 엔진 규칙

- PyChrono, PyDy, SciPy, Rapier는 선택 의존성으로 관리한다.
- 외부 엔진 미설치가 일반 학생용 API 실패로 이어지면 안 된다.
- 무거운 검증은 offline 또는 nightly로 분리한다.
- 외부 엔진 결과는 독립 비교값이며 최종 답 자동 대체값이 아니다.

## API 및 설명 규칙

- 기존 학생용 API 응답 필드는 명시적 migration 없이 삭제하지 않는다.
- 새로운 내부 report는 선택 필드로 먼저 도입한다.
- 풀이 단계는 실제 사용한 식과 모델을 근거로 생성한다.
- 표시용 문자열과 계산용 SymPy 식을 분리한다.
- 오류·모호성·경고를 서로 다른 상태로 표현한다.

## 필수 PR 보고 내용

- 해결하려는 근본 문제
- 변경 파일과 책임
- 새 데이터 계약 또는 API 변경
- 실행한 테스트와 결과
- 성능 변화
- 알려진 한계
- rollback 방법
- 다음 phase에 남긴 작업
