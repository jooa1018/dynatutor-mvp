# Phase 50 integration notes

## Status

Phase 50 adds an offline SymPy Mechanics to SciPy integration layer. It does
not join the production `/solve` critical path and it cannot replace a student
answer. The committed JSON and Markdown reports are runtime artifacts; a
`passed` report is valid only when generated by the Phase 50 runner in the
locked CI environment.

## Data flow

The validation path is:

1. `NumericSimulationSpec` records the model and schema versions, state and
   parameter units, initial conditions, time grid, integration method,
   tolerances, maximum step, events, and optional seed.
2. `build_numeric_typed_model` creates a `TypedDynamicsModel` with explicit
   frames, bodies, forces, constraints, quantities, and display conventions.
3. `build_sympy_mechanics_system` derives a mass matrix, forcing vector,
   energy, observables, and equations of motion with
   `sympy.physics.mechanics.LagrangesMethod`.
4. `run_numeric_system` lambdifies those symbolic expressions, solves the mass
   matrix at each step, and calls `scipy.integrate.solve_ivp`.
5. `NumericSimulationResult` returns the typed and symbolic provenance,
   trajectory, observables, solver diagnostics, versions, invariant drift,
   constraint violation, analytic error, events, warnings, and explicit errors.

The numeric layer is under `backend/engine/simulation/`. Imports are local to
that package; `backend/app/routes/solve.py` and the engine package root do not
import it.

## Supported primary models

### Simple pendulum

- State: `theta`, `theta_dot`.
- Parameters: mass `m`, length `L`, and gravity `g` with explicit SI units.
- Convention: `theta` is measured counterclockwise from the downward vertical.
- Derived equation: `m L^2 theta_ddot + m g L sin(theta) = 0`.
- Diagnostics: mechanical-energy conservation with zero energy at the downward
  equilibrium, fixed-length constraint,
  small-angle analytic trajectory and period, an explicit large-angle
  difference check, and direct integration of the zero equilibrium state.

### Mass-spring-damper

- State: `x`, `x_dot`.
- Parameters: mass `m`, stiffness `k`, and non-negative damping `c`.
- Convention: `x` is measured from spring equilibrium, positive rightward.
- Derived equation: `m x_ddot + c x_dot + k x = 0`.
- Diagnostics: conserved energy when undamped, non-increasing energy when
  damped, and independent homogeneous analytic solutions for underdamped,
  critically damped, and overdamped regimes.

## Safety policy

`phase50-numeric-safety-v1` centralizes the numeric warning and failure limits.
Energy and analytic comparisons use a combined absolute-plus-relative
tolerance computed from the recorded reference scale; constraints use an
absolute residual tolerance because their target value is zero.
The implementation fails closed for an invalid spec, unsupported model,
unavailable SciPy runtime, singular or ill-conditioned mass matrix,
`solve_ivp` failure, non-finite values, and runaway state magnitude. Events are
declared by state variable, threshold, direction, and terminal behavior.

Invariant, constraint, analytic, and stiffness diagnostics are warnings rather
than hidden answer-selection inputs. They remain offline evidence. JSON
serialization replaces no failure with a success and rejects `NaN` or
infinity in the generated report.

## Validation tiers

- Default/fast regression runs short real SciPy trajectories for both primary
  models, symbolic equation checks, invalid-input checks, dependency failure,
  singular-matrix failure, integration failure, non-finite output, runaway
  output, event handling, and production-path isolation.
- Benchmark regression runs seven longer accuracy trajectories: small-angle,
  large-angle, and equilibrium pendulums plus undamped, underdamped, critical,
  and overdamped spring systems.
- `backend/tools/run_phase50_numeric_validation.py` runs the seven-case accuracy
  suite and atomically writes deterministic JSON and Markdown reports without
  timestamps or manually entered result counts.

The runner records the Python platform, SymPy, SciPy and NumPy versions,
integration method, `rtol`, `atol`, `max_step`, evaluation counts, mass-matrix
condition, model versions, schema versions, and safety-policy version.

## Report contract

- JSON: `backend/reports/phase50_numeric_validation.json`
- Markdown: `backend/reports/phase50_numeric_validation.md`
- Report: `phase50-sympy-scipy-numeric-validation-v1`
- Suite: `phase50-numeric-validation-suite-v1`
- Spec/result schema: `1` / `1`

Acceptance requires all seven runtime cases to pass, at least two actual SciPy
trajectories, both primary models, energy and constraint diagnostics, the
applicable analytic comparisons, the intentional large-angle disagreement,
offline-only evidence, no answer overwrite, and byte-identical JSON/Markdown
generation from the same report object.

## Dependency and phase boundaries

SciPy is a required numeric runtime and is already locked by the repository.
PyDy is not imported or required by this layer. Phase 51 owns PyChrono-based
independent validation, and Phase 52 owns the common cross-engine report and CI
tiering; neither is implemented as part of Phase 50.
